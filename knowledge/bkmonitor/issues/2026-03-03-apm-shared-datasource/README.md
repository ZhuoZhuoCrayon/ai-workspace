# APM 支持跨应用共享数据源

## 0x01 背景

### a. Why

当前 APM 采用一应用一数据源的架构，每个应用独占一组 ES 索引。当业务接入大量应用时，ES 索引数量线性膨胀，带来分片数激增、集群元数据开销增大、资源碎片化等问题，严重消耗数据链路资源。

### b. 目标

- **索引收敛**：多应用复用同一数据源，将 ES 索引数量从与应用数等比增长收敛至可控的共享池规模
- **查询透明**：共享模式对上层查询透明兼容，应用间数据通过业务和应用维度严格隔离
- **可扩展**：方案预留日志、指标等数据类型的共享空间，本期聚焦 Trace

## 0x02 实现路线

### a. 建议的方案

**模型与抽象**：将应用与数据源的关系从 1:1 改为 N:1，引入"共享数据源池"概念。独立模型管理共享池的容量（配额与使用量）及元数据，按数据类型可扩展。创建应用时从池中分配使用量最低的可用数据源（不存在则自动新建），删除应用时释放使用量。共享数据源在全局公共空间注册，采用统一序列号命名。

**数据写入**：采集链路在数据清洗阶段从认证令牌中反解业务和应用标识，注入到数据中。无论应用是否使用共享模式，均统一注入，确保数据始终携带隔离维度。

**数据查询**：共享场景下所有查询路径（结构化查询、拓扑发现、全文检索等）统一追加业务和应用维度过滤条件。元数据路由层需支持以业务维度过滤查询全局公共空间的结果表——此为跨团队依赖项，需评估现有能力；若不具备则在 APM 查询层做降级处理，直接指定结果表绕过路由。

**生命周期管理**：现有应用数据源配置层增加对共享池的关联。共享模式下复用池中的数据链路标识，不再自建结果表和索引；启停操作在共享模式下跳过对结果表的直接操作。

### b. 约束

- 本期仅覆盖 Trace 数据类型，日志和指标的共享支持后续迭代
- 不支持存量应用从独占模式迁移到共享模式，仅对新建应用生效
- 元数据路由层对全局结果表按业务过滤的能力为跨团队依赖项，需与 metadata 团队确认
- 共享数据源本期不创建日志索引集
