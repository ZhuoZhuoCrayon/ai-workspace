# APM 支持应用级别配置

## 0x01 背景

### a. 痛点

当前 APM 的服务关联配置（日志关联、返回码重定义等）全部以 `(bk_biz_id, app_name, service_name)` 为粒度，没有「应用级别」或「跨服务共享」的概念。用户想为应用下所有服务设置统一日志关联或统一返回码规则时，只能逐服务手动配置，成本高且易遗漏。

### b. 现状

- 核心模型为 `ServiceBase`（抽象基类），下挂 7 个子类子表。
- 各子表的查询、写入散落在不同调用方，模式不统一。
- 若只改造部分子表，后续扩展时容易重复劳动、遗漏保护逻辑。

### c. 目标

- 支持应用级别（全局）的日志关联和返回码重定义配置。
- 支持指定服务列表作为配置生效范围，而不仅是单个服务。
- 全局配置与服务级配置可共存，服务级优先（覆盖语义）。
- 改造后模式统一，便于维护和扩展。

## 0x02 需求范围

### a. 日志关联（LogServiceRelation）

- Setup 写入、按应用名查询、服务级配置界面展示，均需支持应用级关联。
- 获取关联信息需收口，避免多处自行拼 `(bk_biz_id, app_name, service_name)` 条件。

### b. 返回码重定义（CodeRedefinedConfigRelation）

- List、Set 接口需支持全局与跨服务共享规则。
- 支持按 `service_names` 展开写入，以及按 scope 选择更新策略（服务级 / 全局 / 全量）。
- 全局返回码配置页面仅 RPC 场景需要展示（应用配置模块的业务约束）。

### c. 其他子表

- 除 Log、CodeRedefine 外，`ServiceBase` 下另有 5 张子表，建议一并纳入改造范围，统一查询、写入模式。

## 0x03 设计方向与约束

### a. 全局标识方案

需明确「全局」在数据模型中的表示方式。常见思路包括：


| 方案                                | 说明                                                                      |
| --------------------------------- | ----------------------------------------------------------------------- |
| JSONField `service_names`，空列表表示全局 | 需要评估：JSONField 无法走普通 B-tree 索引；空列表、`null` 在 Python 中有 falsy 陷阱，查询条件易出错。 |
| 显式布尔字段                            | 查询简单、索引友好，需评估对现有 schema 的影响。                                            |
| 特殊 `service_name` 占位符             | 需避免与真实服务名冲突。                                                            |


选型时需兼顾查询性能、向后兼容和可维护性。

### b. 配置下发

- bk-collector 支持 `source="*"` 通配符，表示匹配应用下所有服务。
- 全局规则下发时，可考虑不展开为逐服务枚举，由采集端运行时匹配。

### c. API 行为

- List API 需考虑向后兼容：旧前端不传额外参数时，默认行为应保持不变。
- 返回结果需能区分「全局」与「服务级」，便于前端按场景展示。

### d. 写入保护

- 引入全局记录后，服务级写入需从机制上排除对全局记录的误改。
- 建议统一写入路径，避免多处手写排除条件导致遗漏。

### e. 代码清理

- 部分接口、方法可能已无引用，制定实施方案时需识别并标记待废弃/删除项。
