# 关于 AI 工作区建设的想法

## 0x01 概要

### a. Why

1）AI 作为新一轮的技术革命，借助 AI 将极大提升开发者的产能，增加未来的竞争力。

2）构建一个好用的 AI 分身，需要将原来散落在各平台的文档、代码、笔记等资料进行整合，形成一个结构化的知识库及维护体系，并且能在日常工作中持续更新和完善。

### b. What

1）AI 工作区是下一代协同开发者与模型的工作环境，能让开发者和模型更高效地协同工作。

2）AI 原生知识库：无论是模型还是开发者，都能在 AI 工作区中快速访问和使用各种文档、工具等，提升工作效率和创造力。

### c. How

基于以下设想，请评判方向是否正确，是否有改进空间，同时查找当前的业界最佳实践，给出一些可行的方案：

1）使用场景：
* 个人开发提效：在工作区中与模型进行自然语言交互，由模型辅助完成代码编写、文档撰写、问题解答等任务。
* AI 分身：基于 OpenClaw 构建 AI 分身，能持续学习和更新个人的知识库，成为个人的智能助手。

2）制定一套 AI 工作区的设计原则和规范，确保工作区的可用性、可维护性和可扩展性。
* 面向并兼容 Cursor、Codex、Claude 等主流模型。
* 模型、人类友好：知识索引和交互设计要兼顾模型和人类的需求，确保双方都能高效地使用工作区。
* 便于维护和更新：复杂任务可以需要拆分多次会话或 Agent 协作，工作区需要支持这种复杂的交互模式，能在多 Agent 之间共享单个复杂任务的上下文和状态。

3）模式：
* 多 Git 仓库：探讨一种支持多 Git 仓库的工作区模式，能让开发者在一个工作区中同时管理多个项目的代码和文档。
    * 不同仓库也有自己的 Skills 和 Rules 等，需考虑如何在不同仓库之间共享知识和工具。
    * 例如仓库 A 有自己的 skills 和 rules，工作区中也有一些通用的 skills 和 rules，模型在处理仓库 A 的任务时，能同时访问仓库 A 的知识和工作区的通用知识。
* 统一知识库：
    * 开发需求管理：记录和管理开发需求、任务和进度，形成结构化文档，供模型和开发者随时访问。
    * 问题排查经验
    * 代码片段和工具使用说明等。
