# `README.md` 需求文档写作规则

## 0x01 文档职责

- `README.md` 回答“需求是什么、为什么做、范围是什么、如何验收”。
- 需求文档应承载需求定义、方法论、调研对象列表与非目标。
- 不把调研细节、实现步骤、文件级改造和代码设计塞进 `README.md`。

## 0x02 推荐骨架

| 章节 | 应写内容 |
| --- | --- |
| 背景 / 问题 | 当前痛点、触发条件、业务上下文 |
| 目标 | 本期交付目标与边界 |
| 需求范围 | in-scope 条目 |
| 非目标 | 本期明确不做的内容 |
| 方法论 / 调研路径 | 如何验证需求、要调研哪些对象 |
| 验收标准 | 需求侧完成标准 |
| 参考 / 关联文档 | `PLAN.md`、相关 issue、前置需求 |

## 0x03 写法要求

- 需求条目要能直接映射到实现任务或测试项。
- 非目标要写清楚，避免需求蔓延到实现方案。
- 若需要引用调研结论，用一句话指向 `PLAN.md`，不要在 `README.md` 重复展开。
- 验收标准优先写成可验证句子，例如“边界清晰”“可追溯”“可实现”。

## 0x04 正反案例

### a. 不把实现细节写进需求主干

Bad：

```md
## 0x03 实现方案

- `ServiceBase` 新增 `is_global`
- `ApmMetaConfig` 新增 `code_remarks`
- `ListResource` 支持新参数
```

Good：

```md
## 0x03 需求范围

- 支持应用级统一维护配置。
- 保持服务侧原有读取体验不回退。

## 0x04 非目标

- 本期不展开数据库迁移细节。
- 具体协议与开发落点统一沉淀到 `PLAN.md`。
```

### b. 方法论与验收写成可执行句子

Bad：

```md
我们先调研一下，然后再看看能不能做，最后整理一个方案。
```

Good：

```md
## 0x05 方法论

- 先确认需求边界，再对齐调研对象与已有能力。
- 调研结论统一沉淀到 `PLAN.md`，避免在 `README.md` 重复展开。

## 0x06 验收标准

- 边界清晰：需求范围与非目标可直接执行。
- 可追溯：实现与调研结论可在 `PLAN.md` 中逐项追溯。
```
