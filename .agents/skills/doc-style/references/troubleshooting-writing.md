# `troubleshooting` 写作规则

## 0x01 文档职责

- `troubleshooting` 记录已验证的问题排查经验，帮助后续快速复现、定位和修复。
- 优先服务排障场景，不要把它写成需求背景或长期方案文档。

## 0x02 固定骨架

1. 现象
2. 根因
3. 排查过程
4. 解决方案 / 修复步骤
5. 注意事项 / 回归点

## 0x03 写法要求

- `现象` 写用户可观察到的错误表现、日志、接口返回或异常行为。
- `根因` 写经验证的真正原因，不写猜测。
- `排查过程` 按时间或因果顺序组织，保留关键命令、关键日志和转折点。
- `解决方案 / 修复步骤` 写成可复用动作列表，必要时补充回滚或兜底方案。
- `注意事项 / 回归点` 只保留后续最容易遗漏的风险点。

## 0x04 结构化承载

- 命令、SQL、PromQL、curl 等放进代码块，并标注语言类型。
- 多个检查点优先用有序列表。
- 多个现象、根因或风险优先用表格对齐。

## 0x05 正反案例

### a. 先收敛关键信息，再展开修复

Bad：

```md
这个问题查了很久。
一开始看接口报错，后来又去翻日志，再去看代码。
最后发现是空值没处理。
改一下判空应该就可以。
```

Good：

```md
## 0x01 关键信息

### a. 现象

- 详情接口返回业务失败。
- 异常直接落在某个清洗函数的空值访问上。

### b. 根因

- 上游允许返回 `None`。
- 下游未判空，仍继续访问字段。

## 0x02 解决方案

- 增加空值兜底。
- 保持原有语义不变，只修复接口不应崩溃的问题。
```

### b. 排查过程写成可复用步骤

Bad：

```md
排查时看了接口、日志、代码。
最后就知道是这里有问题。
```

Good：

```md
## 0x03 排查过程

1. 先复现接口报错并记录异常位置。
2. 再确认上游返回是否允许为空。
3. 最后核对下游是否缺少判空或默认值兜底。
```
