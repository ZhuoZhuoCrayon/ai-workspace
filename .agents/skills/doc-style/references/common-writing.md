# 通用 Markdown 写作规则

## 0x01 总体原则

- 结论先行：先写判断，再补必要说明。
- 规则表达写成可执行约束，不写空泛口号。
- 每句话只表达一个信息点，超过两层逻辑就拆句。
- 每段只聚焦一个主题，单段不超过 `3` 句。
- 解释类内容优先改写成列表、编号或表格，让读者在 `10` 秒内抓住要点。

## 0x02 标题与结构

| 主题 | 规则 |
| --- | --- |
| 标题层级 | 不跳级，且一级标题下不要直接进入三级标题。 |
| 标题命名 | 只表达当前小节主题，且同级标题保持同一颗粒度。 |
| 章节编号 | 需要编号时优先使用 `## 0x01`、`### a.` 这类稳定格式。 |
| 章节组织 | 先写总思路与关键结论，再展开模型、查询、写入、验证、风险等细节。 |

## 0x03 句子与段落

| 主题 | 规则 |
| --- | --- |
| 句子密度 | 只保留有效信息，避免“进一步优化”“增强体验”这类方向词。 |
| 段内限制 | 同一 Markdown 段落或列表项内不要堆叠多个长句，优先拆分。 |
| 单行写法 | 单句合并后不超过 `120` 个字符时，写成一行。 |
| 风格禁忌 | 避免“简单来说”“这里先说明一下”这类口语化和自述式表达。 |

## 0x04 列表与表格

| 载体 | 何时使用 | 规则 |
| --- | --- | --- |
| 无序列表 | 并列结论、约束、验证点 | 每个列表项只承载一条信息。 |
| 有序列表 | 多步操作、流程 | 每步只写一个动作，不用分号串联。 |
| 表格 | 字段固定且项数超过 `1` | 方案比较、规则条件、风险分级、文件归属优先用表格。 |

表格单元格补充规则：

- 不用分号串联多项内容。
- 多项内容优先用 `[1] xxx<br />[2] xxx`。
- 需要脚注时使用 `*[1]*`、`*[a]*` 这类斜体角标。

## 0x05 链接与微观格式

| 主题 | 规则 |
| --- | --- |
| 链接文案 | 使用摘要文案，不使用裸 URL。 |
| 引用 | 引用外部资料时给出来源或链接，避免孤立结论。 |
| 中英文间距 | 中英文之间加空格。 |
| 标点 | 中文语境优先使用全角标点。 |
| 代码标识 | 标识符用反引号包裹，代码块标注语言类型。 |
| 源码链接 | 使用统一前缀：`<源码>` + 项目名 + 文件路径。 |

## 0x06 文件级约束

- 文件命名使用小写英文加短横线，例如 `redis-connection-timeout.md`。
- 索引文件统一命名为 `INDEX.md`。
- 文件末尾保留一个空行。
- 默认作用于工作区自有 `*.md` 文档。
- 第三方目录与依赖目录不纳入本规则。

## 0x07 正反案例

以下案例提炼自优秀方案文档的共性特征。

### a. 结论先行

Bad：

```md
这个方案会涉及共享、隔离、命名、查询和写入几个方面。
下面分别展开。
```

Good：

```md
本期方案拆成两条线：

- **共享池管理**：负责容量与元数据。
- **应用侧引用**：通过关联字段复用共享池。

关键结论：

- **职责分离**：共享池不负责外部链路创建。
- **查询隔离**：业务隔离与应用隔离分别在不同层处理。
```

### b. 先抽象规则，再补细节

Bad：

```md
先改三个接口，再补两个字段，最后补查询过滤。
```

Good：

```md
关键结论：

- **展示优先级固定**：读取顺序按既定层级合并。
- **旧协议兼容**：服务视角继续输出原有结构。

开发落点放到后文文件级变更表，不在主干小节提前展开。
```
