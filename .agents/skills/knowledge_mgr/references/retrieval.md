# Retrieval Guide

只在检索场景阅读本文件；纯写操作不要预加载本文件。需要确认稳定检索优先级时，再补读 [retrieval-rules.md](retrieval-rules.md)。

## 0x01 适用场景

当任务属于以下任一类时，读取本文件：

- 回答依赖历史知识资产的项目问题
- 继续某个知识对象目录的既有上下文
- 编码前核对既有方案、排障结论或代码片段
- 核对 public / private 知识是否一致
- 为写操作补齐前置证据

## 0x02 检索契约

1. 输入上下文至少包含：`project`、当前任务目标、相关 issue 路径（如存在）和已知约束。
2. 按 `RULE-KNOWLEDGE-002` 自上而下检索；除会话上下文外，默认并行检查 public / private 两侧知识。
3. 输出证据至少包含：`路径`、`命中理由`、`updated` 或最近更新时间。
4. 若命中对象为 `snippet`，默认直接给出与当前任务最相关的代码块、命令块或关键片段，并解释其用途、输入参数和执行效果；不要只返回路径。
5. 若命中对象为 `troubleshooting`，默认直接给出“现象 → 根因 → 排查过程 → 解决方案 / 注意事项”；路径与更新时间作为辅助证据放在后面。
6. 若证据冲突，先列出冲突点，再决定是否继续下钻。
7. 只有当前层证据不足时，才进入下一层。
8. 进入源码前，必须明确说明知识未命中或知识冲突的原因。

## 0x03 常见检索模式

### a. 继续知识对象目录

1. 先读当前知识对象目录下的 `README.md` 与 `PLAN.md`。
2. 再查同项目 `knowledge/<project>/` 与 `private/knowledge/<project>/`。
3. 若仍不足，再进入项目规范与源码。

### b. 项目知识问答

1. 从项目知识库起步，而不是直接进入源码。
2. 若问题显式要求“代码片段 / 需求 / 方案 / 排障”，优先命中对应分类目录。
3. 命中 `snippet` 时，优先贴最相关代码块并说明“这段代码在当前问题里怎么用”。
4. 命中 `troubleshooting` 时，优先贴排查 / 修复流程，必要时附关键代码或命令。
5. 回答时回带证据路径，避免凭记忆下结论。

### c. public-private 冲突核对

1. 并行读取两侧命中项。
2. 先标出冲突字段：标题、摘要、更新时间、结论。
3. 若冲突无法在知识层解决，再补充项目规范或源码证据。
