# Operation Rules

只在需要确认 issue / PLAN 稳定规则原文时阅读本文件。

## 0x01 操作前提

**【CRITICAL（必须执行不可协商）】任何知识资产写操作完成后，都必须执行后置检查。**

## 0x02 RULE-ISSUE-001 — Issue 目录结构

- Issue 目录只使用 `README.md` 与 `PLAN.md` 两个文件。
- 任务进展统一写回 `PLAN.md`，不创建 `PROGRESS.md`。

## 0x03 RULE-ISSUE-002 — 迭代交付写回

基于 issue 的迭代在出现实质变更（代码、文档或配置修改）后，交付前必须主动询问是否同步 `PLAN.md`。

- 问法应贴合当前 `PLAN.md` 的实际结构，不强制绑定固定章节编号。
- 重点是确认是否将本轮有效结论回写到受影响的方案主干片段，并补充实施进展表。
- 若用户拒绝可跳过，但下一轮仍需再次询问。

## 0x04 RULE-ISSUE-003 — Issue 文档职责分离

| 文件 | 应写内容 | 不写内容 |
| --- | --- | --- |
| `README.md` | [1] 需求定义。<br />[2] 方法论。<br />[3] 调研对象列表。 | [1] 调研细节。<br />[2] 对比表结论。<br />[3] 语法样例结论。 |
| `PLAN.md` | [1] 调研结论。<br />[2] 对比分析。<br />[3] 实现路径。<br />[4] 验收。<br />[5] 进展。 | 不把需求定义主干重新写成 `README.md` 的副本。 |

- 若发现 `README.md` 混入调研细节，必须先回收至 `PLAN.md` 再继续。

## 0x05 RULE-ISSUE-004 — PLAN 迭代模式

`PLAN.md` 的迭代必须采用“方案主干实时更新 + 进展表格化”模式，禁止写成流水账，或仅以补丁式追加替代主干更新。

### a. 方案主干

| 主题 | 规则 |
| --- | --- |
| 记录视角 | `PLAN.md` 应从架构师视角记录当前有效方案、关键约束、交付路径与验收标准。 |
| 细节边界 | 不展开具体实现代码，最多保留少量伪代码。 |
| 更新顺序 | 执行过程中出现新发现时，先定位受影响的设计片段并更新其内容。 |
| 主干目标 | 只读方案主干也应能获得当前有效方案。 |

### b. 实施进展

| 主题 | 规则 |
| --- | --- |
| 表头 | 实施进展记录在统一表格中，建议表头为 `时间`、`对应设计片段`、`结论调整概要`、`改动 / 验证`。 |
| 时间格式 | `时间` 列精确到小时，统一使用 `` `YYYY-MM-DD HH:00` `` 格式。 |
| 合并原则 | [1] 同一天持续打磨同一方案，且未形成新的阶段结论时，必须合并为一条进展。<br />[2] 只保留当天最终有效结论。 |
| 新增条件 | 只有出现新的阶段里程碑、独立阻塞或独立验证结论时，才新增一行进展。 |
| 记录内容 | 进展表只写本次新增事实、当前结论与下一步动作，不重复抄写方案正文。 |
| 写回顺序 | 若新进展改变原方案结论，先更新方案主干，再更新进展表。 |
| 单元格格式 | 表格单元内用 `<br />` 或 `[1]`、`[2]` 角标换行，禁止松散段落。 |
| 膨胀处理 | 若单元格内容开始膨胀，先更新方案主干或拆分主题，不把长内容塞进进展表。 |
| 版本锚点 | 版本锚点只记录 `分支` 与 `PR`。 |
| frontmatter | 每次写回后同步 frontmatter `updated` 为当日日期。 |

Bad：

```md
| 时间 | 结论调整概要 |
| --- | --- |
| `2026-04-16 10:00` | 调整表格文案 |
| `2026-04-16 11:00` | 再调整表格文案 |
| `2026-04-16 14:00` | 继续调整表格文案 |
```

Good：

```md
| 时间 | 结论调整概要 |
| --- | --- |
| `2026-04-16 14:00` | [1] 收敛当天最终方案<br />[2] 合并重复进展并完成文档检查 |
```

迭代收口流程：

1. 简短复盘：本轮做了什么、修了什么、验证了什么。
2. 按 RULE-ISSUE-002 主动询问用户。
3. 用户确认后，先更新受影响的方案主干片段，再将进展写入表格，同步 `updated` 日期。
